# product · 产品 / 功能文档

> 记录用户能做什么、功能规则是什么、当前支持哪些能力。
> 系统架构看 `../architecture/`，API 文档看 `../api/`，开发进度看 `../dev-phase/`。

本目录只描述系统对用户、运营和管理员呈现出的产品能力，不写工程实现细节。

维护原则：

- 产品文档永远描述“当前系统长什么样”，只沉淀已实现或确定落地的能力。
- 未开工、未确认或只是想法的设计不要写进 product；需要时先放到 `docs/plan/`，开工后进 `docs/dev-phase/`，完成后再回填 product。
- 每篇文档聚焦一个业务域，优先写产品目标、核心对象、关键规则、权限/隔离边界、后台运营能力和当前状态。
- 避免把接口路径、函数名、类名、SQL 细节作为主体内容；这些分别属于 `docs/api/`、代码和架构文档。
- 改了功能规则、用户可见行为、权限边界、数据归属或业务状态时，必须同步更新对应 product 文档。
